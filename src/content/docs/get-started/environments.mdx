title: "Environments & testing"
description: "One production environment, one live API key, and a welcome credit pack so you can test the integration end-to-end with real provider games before funding your wallet."
---

The Aggregator has **one production environment**. There is no separate "test" or "sandbox" key tier — you get one `sk_live_*` API key when you register, and you use it for both integration testing and live traffic.

To make pre-launch testing safe, every new operator account is provisioned with a **welcome credit pack** — a small pool of real-money credits that the platform funds. You can use those credits to launch real provider sessions, receive real wallet callbacks, and exercise the entire money path without funding your own wallet first.

When the welcome pack runs out, you fund your operator wallet with credits the normal way (in-cabinet purchase or invoice billing) and continue.

## Base URL

```
https://api.aggregator.gg/v1
```

All operator integrations use this single base URL. Pre-launch testing uses **welcome credits** + **demo sessions** (see below) — not a separate test cluster.

## How welcome credits work

1. You register your operator account in the [cabinet](https://app.aggregator.gg/register).
2. The platform team activates your account and credits the welcome pack (typically tens to hundreds of real spins, depending on launch agreement).
3. Your `sk_live_*` key processes real-money sessions immediately.
4. Bets are charged against your operator wallet; the welcome pack subsidises that wallet until it is depleted.
5. Once welcome credits are used up, you continue funding the wallet through the normal billing flow.

The platform does not distinguish "welcome traffic" from "paid traffic" anywhere in the API surface — every session, transaction, and callback looks identical regardless of which credit source is paying. You do not need to flag test traffic.

## Demo sessions — no money at all

If you want to launch a game purely to inspect the UI or render a preview iframe without any wallet movement, use [`POST /v1/demo-sessions`](/api/v1/operations/createdemosession/) instead of `/v1/sessions`. Demo sessions:

- Do not debit any wallet
- Do not emit `bet` or `win` callbacks
- Have no impact on your welcome credit balance

Use demo sessions for marketing previews, lobby thumbnails, or showing a game inside your cabinet without any money flow.

## Welcome-pack best practices

- **Treat it like production from day one.** The same money-path code that handles a welcome-pack spin will handle your real customer's first deposit. Do not branch on "is this a test bet?" anywhere — the API does not give you that distinction, and code that branches on it ages badly.
- **Run your full callback verification path.** Wallet callbacks during welcome-pack testing are real HMAC-signed callbacks from the real provider. If your signature verification has a bug, you find out now, not after you go live.
- **Reconcile against the [transactions API](/api/v1/operations/listtransactions/).** Before you exhaust the welcome pack, confirm your wallet ledger matches what the platform records. Any drift discovered now is much cheaper to fix than the same drift after real-money traffic starts.

## Rate limits

Default rate limits, evaluated in a 60-second sliding window:

- **600 requests/minute per API key**
- **3 000 requests/minute per operator account** (sum across all keys)
- **100 failed authentication attempts/minute per IP** (brute-force protection)

Exceeded limits return `429 Too Many Requests` with `X-RateLimit-*` headers and `Retry-After`. See [Authentication](/get-started/authentication/) for full header semantics.
